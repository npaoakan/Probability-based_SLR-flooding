# Static Sea-Level Rise (SLR) Inundation Modeling for the Hawaiian Islands

A probability-based static SLR inundation model that produces flood depth rasters and extent shapefiles for each of the main Hawaiian Islands at 1-foot SLR increments from 0 to 10 ft. Built with ArcPy, NumPy, and SciPy in the ArcGIS Pro Python environment.

Author: Noah Paoa. Adapted/Inspired from NOAA's SLR Viewer Mapping Confidence script with the help of Tiffany Anderson. Updated for Python 3 and ArcGIS Pro on 04/01/2025.

---

## What the Model Does

For every pixel of a digital elevation model (DEM), the script asks: *given the uncertainty of the DEM elevation and the uncertainty of the water surface, what is the probability that this pixel is inundated at a given SLR increment?* Pixels above a chosen confidence threshold (80% by default) are classified as flooded. Flooded pixels are then split into two classes:

- **Surficially Connected Inundation (SCI)** — flood polygons that touch the coastline (i.e., a continuous water path to the ocean).
- **Topographically Isolated Inundation (TII)** — flood polygons that do not touch the coastline. These are areas low enough to flood at the given water level but disconnected from the ocean by surrounding higher ground and based on Habel et al.'s findings can be expected to be inundated by groundwater.

Flood depths are reclassified into 1-ft depth bands and saved as low-bit-depth GeoTIFFs to keep file sizes manageable.

This workflow is intended for Hawaiʻi specifically because VDatum is not available there; TCARI is used as the tidal datum surface instead.

---

## The Probability-Based Inundation Formula

Combined vertical uncertainty:

```
sigma_d = sqrt(dem_uncertainty^2 + water_uncertainty^2)
```

Per-pixel inundation depth at probability `prob`:

```
d_tilda = (waterSurface - dem) + sigma_d * sqrt(2) * erfcinv(2 * prob)
```

A pixel is considered flooded when `d_tilda >= 0`. `erfcinv` is the inverse complementary error function from `scipy.special`. `waterSurface` is the TCARI MHHW raster plus the SLR increment.

`prob = 0.8` (80% confidence that the pixel is wet) is used throughout both the coastline-creation and passive-SLR-modeling sections of the script.

## Inputs Required

- **DEM** (per island) and its vertical uncertainty (one number per island).
- **Water surface** — TCARI MHHW raster — and its vertical uncertainty (or a single MHHW value with its standard deviation).
- **Coastline shapefile** for ocean-connectivity analysis (generated by the first cell of the script the very first time you run it; see workflow below).
- **SLR increments** to model.

### Uncertainty values used in the script

| Component | Value | Source / notes |
| --- | --- | --- |
| TCARI | 0.073 m | RSS of RSS errors at the location of the tide gauge |
| Default DEM uncertainty | 0.2 m | applied to all islands not listed below |
| Oʻahu DEM uncertainty | 0.3 m | used in coastline cell |
| Kahoʻolawe DEM uncertainty | 0.62 m | |
| Niʻihau DEM uncertainty | 0.56 m | |

If you are unsure about a DEM uncertainty value for a new dataset, check with whoever produced the DEM.

### MHHW datum offset

NOAA defines MHHW relative to a fixed tidal epoch. Two offsets are encoded in the script to bring the surface forward to a chosen reference year:

- `+0.02 m` for displacement of MHHW from 1992 to 2000
- `+0.04 m` for displacement of MHHW from 1992 to 2005

## Script Sections

The notebook (`Static_SLR_Modeling.ipynb`) contains two main sections that run sequentially; Create Coastline & Passive SLR Modeling. SLR modeling requires a coastline shapefile.

### 1. Create Coastline

Run this section **once per island** to generate the coastline shapefile (at MHHW) that all later steps depend on. **Unnecessary to re-run if you already have a valid coastline.** The cell:

1. Loads the island DEM and the TCARI MHHW raster.
2. Computes `d_tilda` at 0 ft SLR (with the 0.02 m MHHW offset) using the probability formula at `prob = 0.8`.
3. Reclassifies all pixels with `d_tilda >= 0` to value 1 (flooded). Kahoʻolawe uses a slightly different reclassification (`Con(IsNull(u_dem), 1, Con(u_dem >= 0, 1))`) to handle edge effects of the specific DEM we have.
4. Runs `RegionGroup` with the EIGHT-neighbor / CROSS rule to find connected pixel clusters.
5. Identifies the largest region (the ocean) by counting pixels per region group.
6. Reclassifies the ocean to 1 and everything else to NULL and converts the result to a polygon shapefile representing the ocean and TII inundation (`<island>_80prob_MHHW_coastline.shp`).

**Manual cleanup is required after this step.** Open the resulting shapefile in a GIS software and:

- Keep the polygon representing the ocean (and any inland water features that have an underground connection to the ocean; e.g., the Natatorium).
- If the DEM contains NoData pixels, they might show as artificial land areas. Remove them.

If the post-cleanup coastline shapefile is identical to the 0 ft flood map, the next section might fail with `ERROR 001566: Failed to create raster dataset. The clip feature is outside the raster extent.` If the error is raised, is because the extent of flood at 0 ft is identical to the coastline shp.

### 2. Passive (Static) SLR Modeling

For each island and each SLR increment, the script:

1. Computes `d_tilda` from the DEM, TCARI surface (+ SLR), and combined uncertainty.
2. Reclassifies `d_tilda >= 0` to 1 (flooded), runs `RegionGroup`, and converts the result to a polygon shapefile of flood extents.
3. **Splits flood polygons into TII vs. SCI** using `SelectLayerByLocation` with INTERSECT against the coastline. Polygons that do not intersect the coastline are TII; the remainder are SCI.
4. **Filters small TII clusters.** Computes polygon area (`AddGeometryAttributes`, square meters) and deletes TII polygons with area ≤ 100 m². Area is enforced as the threshold rather than pixel count, so the cutoff is independent of cell size.
5. **Builds the SCI raster** by erasing the ocean from the flooded area (`Erase_analysis`) and clipping `d_tilda` to the result.
6. **Builds the TII raster** by clipping `d_tilda` to the cleaned TII shapefile.
7. **Reclassifies depths** into eleven 1-ft bands (in meters: 0–0.3048, 0.3048–0.6096, …, 2.7432–3.048, plus a >3.048 m bin), with all values ≤ 0 set to NoData. Output is written at 4-bit depth via `CopyRaster_management` to keep file sizes small.
8. **Builds the total inundation extent shapefile** by merging the TII and SCI shapefiles. At the 0 ft increment, only TII exists (because at 0 ft SCI equals the coastline itself), so total inundation is just TII. If no TII areas survive the area filter, total inundation is just SCI (e.g., no flooding).
9. **Builds raster attribute tables** for the final 4-bit rasters.
10. **Cleans up.** Force-runs `gc.collect()`, clears the ArcPy workspace cache, and empties the temp folder before moving to the next SLR increment — otherwise builds tend to crash partway through. You might get several errors as some files in the temp folder are still held in memory and can't be deleted. These errors can be disregarded.

## Output Structure

For each island and SLR increment, the script writes:

- `Outputs/<island>/SCI/Unclass/<island>_80prob_<XXft>_SCI.tif` — continuous flood depth, surficially connected.
- `Outputs/<island>/SCI/<island>_80prob_<XXft>_SCI.tif` — reclassified into 1-ft depth bands, 4-bit.
- `Outputs/<island>/TII/<island>_80prob_<XXft>_TII.tif` — reclassified TII flood depth, 4-bit.
- `Outputs/<island>/Extent_shp/<island>_80prob_<XXft>_totalI.shp` — merged TII + SCI extent shapefile (or just one of the two if the other is empty).

The Oʻahu notebook additionally writes:

- `Outputs/Oahu/DBI/HI_Oahu_80prob_<XXft>_DBI.tif` — drainage backflow inundation depth raster, 4-bit.
- `Outputs/Oahu/StormDrains/HI_Oahu_80prob_<XXft>_stormDrain_intersect.shp` — storm-drain points that intersect inundation at that SLR level.
- `Outputs/Oahu/Roads/1ft/Oahu_80prob_<XXft>SLR_1ft_floodedStreet.shp` — road segments flooded by ≥ 1 ft.
- `Outputs/Oahu/Roads/2ft/Oahu_80prob_<XXft>SLR_2ft_floodedStreet.shp` — road segments flooded by ≥ 2 ft.

## Running the Script

### Prerequisites

- ArcGIS Pro (with the Spatial Analyst extension license).
- A geodatabase. The script needs one as a placeholder workspace even though it is not used to store outputs. To create one:
  1. Open ArcGIS Pro → Catalog → New Project, point at the folder you want.
  2. Name the project; the geodatabase will be created as `<project_name>.gdb`.
  3. Update the path in the notebook to match (`arcpy.env.workspace`).
- A Python environment cloned from the ArcGIS Pro environment (referred to as `arcgis_env` in the notebook header).
- Directories that match the input and output paths.

### Order of operations

1. Run the **Create Coastline** cell at 0 ft SLR / 80% probability.
2. Manually clean the resulting coastline shapefile (see step 1 above).
3. Run the **Passive SLR Modeling** cell over the islands and SLR increments you want.

### If the script crashes mid-run

The script is designed to be re-runnable; `arcpy.env.overwriteOutput=True` is set so existing outputs will be overwritten.

## Oʻahu-Specific Workflow

A separate notebook (`Static_SLR_Modeling_Oahu.ipynb`) is provided for Oʻahu. It contains the same Create Coastline and Passive SLR Modeling sections as the main script, plus two Oʻahu-specific extensions: drainage backflow inundation and flooded streets.

### Additional inputs

- **Storm drain shapefile** (`Storm_Water_-_Structure_EPSG6634.shp`) — point features representing storm-drain structures across Oʻahu.
- **Roads shapefile** (`tl_rd22_hi_roads_filtered_EPSG6634.shp`) — line features representing the road network.

### Drainage Backflow Inundation (DBI)

After SCI and TII have been computed and merged into the total inundation shapefile for a given SLR increment, the Oʻahu notebook adds a DBI step inside the same loop:

1. Splits the total inundation shapefile into single-part polygons (`MultipartToSinglepart`).
2. Selects only the polygons that intersect a storm-drain point (`SelectLayerByLocation` with INTERSECT against the drainage shapefile).
3. Saves the selected polygons as a DBI flood shapefile.
4. Saves the subset of storm drains that intersect those polygons to `Outputs/Oahu/StormDrains/HI_Oahu_80prob_<XXft>_stormDrain_intersect.shp`.
5. Clips `d_tilda` to the DBI flood shapefile, reclassifies into the same 1-ft depth bands, and writes a 4-bit raster to `Outputs/Oahu/DBI/HI_Oahu_80prob_<XXft>_DBI.tif`.

DBI is therefore a *subset* of the total inundation (SCI + TII after the ≤100 m² TII size filter) — specifically, the polygons that contain or touch a storm drain. The interpretation is that these are the areas where flooding can be amplified by, or attributed to, compromised drainage / storm-drain backflow rather than purely surface or groundwater pathways. If no inundation polygons intersect any storm drain at a given SLR increment, the DBI step is skipped for that increment.

The same 100 m² area filter from the main script applies (i.e., DBI inherits the cleaned TII shapefile, not the raw one).

### Flooded streets

After all SLR increments have been processed, a separate cell identifies streets flooded by ≥ 1 ft and ≥ 2 ft of standing water at each increment:

1. **Builds a full flood raster** for each SLR increment by mosaicking the SCI and TII reclassified rasters (`MosaicToNewRaster`). At 0 ft, the TII raster alone is used (since SCI is empty at 0 ft).
2. **Reclassifies the full flood raster** to keep only pixels in depth band ≥ `feet` (1 or 2), setting everything else to NoData.
3. **Converts the result to polygons** and **intersects** them with the roads shapefile (`Intersect`, LINE output), producing line segments of road that fall inside the flooded area at the chosen depth.

Outputs are written to:

- `Outputs/Oahu/Roads/1ft/Oahu_80prob_<XXft>SLR_1ft_floodedStreet.shp`
- `Outputs/Oahu/Roads/2ft/Oahu_80prob_<XXft>SLR_2ft_floodedStreet.shp`

The depth thresholds are independent of SLR — they describe how deeply a street is flooded *at* a given SLR level (e.g., the 1 ft / 2 ft criteria are sometimes used as proxies for "passable by most vehicles" vs. "impassable").

## Common Errors and Fixes

**`ERROR 001566: Failed to create raster dataset. The clip feature is outside the raster extent.`**
The coastline shapefile is identical to the 0 ft flood extent (manual cleanup of the coastline was skipped or incomplete) or its outer boundary doesn't extend past the DEM. Re-do the manual cleanup of the coastline so SCI has something to clip against, and extend the seaward boundary well past the DEM edge.

**`ERROR 999999: Error executing function. Unexpected operation [GDB_Items]. No spatial reference exists. Failed to execute (Reclassify).`**
The geodatabase is corrupted (this has been observed after a forced restart mid-run). Delete the `.gdb` and create a new one with the same name; outputs will align with the DEM again.

**`ERROR 99999: Failed to execute Function.`**
Trigger unknown; historically resolved by swapping to a fresh dummy geodatabase.

**Crashes from too many files in temp**
The script empties `temp_path` at the end of every SLR increment for exactly this reason. If you customize the script, keep that cleanup in place.

## Glossary

- **SCI** — Surficially Connected Inundation. Flooding with a continuous water path to the ocean.
- **TII** — Topographically Isolated Inundation. Low-lying inland flooding disconnected from the ocean, corresponding to groundwater inundation in this model.
- **DBI** — Drainage Backflow Inundation. Subset of total inundation (SCI + cleaned TII) consisting of polygons that intersect storm-drain points (Oʻahu-specific).
- **TCARI** — Tidal Constituent And Residual Interpolation. Used here in lieu of VDatum, which is not available for Hawaiʻi.
- **MHHW** — Mean Higher High Water. The tidal datum used for the water surface.
- **MSL / LMSL** — Mean Sea Level / Local Mean Sea Level. The reference datum the DEMs are tied to.
- **DEM** — Digital Elevation Model.
